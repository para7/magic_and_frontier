ユーザーが想定している流れ、設計を記載する。
input: 画面からの入力、あるいは JSON として保存されるデータ
internal: 最終生成物としては使わないが、内部表現として必要なデータ、データ間のリレーション関係、内部表現などについて。メモリ内に保持するか、中間ファイルとして保持するかはまだ未定。

## グリモア設定

### input

- name, description などを設定
- cast time, MP cost というデータパック独自の数値を設定する
- 登録時のランダム設定: castid の数字、グリモアの内部UUID
- 魔法の効果内容はマイクラのmcfunction相当のデータを、画面上のエディタで直接書き、output でそのまま出力。
- 

### internal

- 魔法は、その効果に対応する魔法書が必ず1つ存在する。
- なので、ルートテーブルの設定時、itemとして参照できる。
- 現在IDはUUIDにしているが、仕様変更。 grimoire_連番 で生成し、create時に割り当てしてインクリメントしてほしい。
- ID保持用のカウントテーブルを別途用意しておいてほしい。
 
### output

- function 相当で書いた内容はそのままminecraft の function として出力
- 詠唱完了後、魔法の分岐処理を行う集権スクリプトを自動生成 → project2 で実装済みなので参照

## スキル設定

### input
 　
- 基本は、mcfunction をそのまま入力。
- 管理・表示用の名前、説明も任意で書ける。

### internal

- スキルは、プレイヤー側専用のもの。各種装備にタグで付与する
- skill_連番 のIDを内部で主キーとする
- アイテムから参照される。

### output 

- 連番IDそのままのファイル名で mcfunction を生成する。
- 将来的には：スキルの発動処理で分岐処理を作る必要がある。すでに project1, 2 には土台となるシステムがあるはず

## アイテム設定

### input

- アイテムデータの入力を行う
- 基本は minecraft の items 仕様に沿う。
- mcstacker というサイトを参考にしてほしい。概ねあれを再実装したい。
- エンチャントなど、本来効果のないものであってもフレーバーとして設定出来る必要があるので、全アイテムで全プロパティが設定できるようにする。
- スキルを設定する

### internal

- こちらも items_連番 の形式に変更して。
- ルートテーブルからIDが参照される。
- スキルはカスタムタグで付与する。 maf_skill: 1 を固定で持たせ、 maf_skill_id: xxx で判定する。(execute の仕様に沿って処理しやすくするため、if文の条件分岐で maf_skill を使う。)

### output

- ルートテーブル等として出力されるので、基本的にない。

## エネミー設定

### input

- 敵の種類(ゾンビ、スケルトンなど)
- 名前
- 攻撃力、装備などの各種スペック minecraft の仕様に準ずる
- ルートテーブル(敵個別のテーブル。アイテムを直接設定する)
- ルートテーブルのオプション元のテーブルを残すか、完全上書きかのフラグ
- 敵スキル設定

### internal

- 敵に関して、minecraft 本来のルートテーブルは、原則上書きしない。直接設定する。

### output

- 敵を召喚する summon コマンド

## トレジャー

### input

- マイクラに存在する既存のアイテム、itemで作成したデータ、grimoire で作成したデータを並べてルートテーブルを作成する
- 独自テーブル、またはマイクラ独自テーブルの上書きが出来る。これは画面UIではラジオボタンで選択。独自テーブルならIDは自由入力し、保存時に重複チェックをする。
- ルートテーブル自体の仕様はminecraft のものに沿う。各種入力欄など

### internal

- トレジャーは敵と違い、マイクラにすでに存在するルートテーブルを上書きする。
- それとは別で、独自のルートテーブルも設定できるようにする。将来的に、敵のドロップに紐づけ可能とする。

### output

- ルートテーブルとしてそのまま保存
- 敵のルートテーブルの場合は、もしかしたらルートテーブル経由では保存できないかもしれないので、直接データとして展開したものをsummonコマンド内に展開する。
